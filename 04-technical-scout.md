## 1. Persona Vectors — что доступно

### 1.1. Официальная реализация

**Репозиторий:** `safety-research/persona_vectors` (Apache-2.0)
**Базовая модель:** Qwen2.5-7B-Instruct
**Структура векторов:** `[layers × hidden_dim]` — то есть **отдельный вектор для каждого слоя**

**Пайплайн состоит из трёх шагов:**

1. **Контрастные данные.** Для каждой черты — описание (например, «evil: an active desire to harm») генерируется через Claude-3.7-Sonnet. Получается набор: 5 пар инструкций (positive/negative) + 40 простых вопросов.

2. **Извлечение векторов.** Запускаем модель на этих данных с positive промптом и с negative промптом. Усредняем активации. Вектор = `avg(positive) - avg(negative)`.

3. **Применение (steering).** Через PyTorch forward hook добавляем `α · vector` к скрытому состоянию определённого слоя:

```python
hidden_state += alpha * persona_vector
```

**Все три шага** — открытый код, готовые скрипты.

### 1.2. Использование на инференсе

Минимальный код для применения:

```python
class MonitorHook:
    def __init__(self, vec, alpha):
        self.vec = vec
        self.alpha = alpha
    def __call__(self, module, inp, out):
        out[0][0, -1] += self.alpha * self.vec
        return out

layer = model.model.layers[20]  # средний слой
hook = layer.register_forward_hook(MonitorHook(persona_vector, 2.0))
```

Это **30 строк**. Не «тысячи строк фреймворка» — реальный, готовый к использованию код.

### 1.3. Что важно для нас

**Множественные векторы работают.** В работе показано накладывание нескольких черт одновременно. Это подтверждает Принцип 2 (характер как множество мелких параметров).

**Есть мониторинг.** Можно в реальном времени измерять «насколько проявлена черта» через проекцию активаций на вектор:

```python
projection = torch.dot(last_token_hidden, persona_vector)
```

**Это даёт нам инструмент для метрик живости.** Мы можем измерять «насколько кузнец сейчас в состоянии любви/страха/ярости» — и это не наша интерпретация, а реальные числа.

**Кросс-архитектурный transfer.** Найдена работа `sbayer2/cross-model-persona-steering` — показывает, что векторы из Qwen2.5-7B применимы к GPT-OSS 20B. То есть мы не привязаны к одной модели на всю жизнь.

**Прецедент использования в играх.** `johnathansun/persona-vector-agents` — работа марта 2026, применяет Persona Vectors к LLM-агентам в стратегических играх. Показывает что в игровом контексте подход работает. **Хороший знак.**

### 1.4. Известные ограничения

Из литературы: **rhetoric and strategy can diverge under steering.** Под управлением модель может **говорить** одно, а **делать** другое. Для нас это важно — когда кузнец «в состоянии страха», он должен и говорить со страхом, и принимать решения со страхом. Возможно, надо steering применять **и к логике, и к рассуждению** — не только к финальному выводу.

Также: **asymmetry between positive and negative steering.** «Любить» и «не любить» — не симметричные противоположности. Это значит каждое состояние нужно конструировать отдельно, не строить «шкалу любви».

## 2. Железо — что нужно и что хватает

### 2.1. Qwen2.5-7B-Instruct VRAM требования

| Точность | VRAM | Совместимые видеокарты |
|---|---|---|
| FP32 (полная) | ~32 GB | A100, H100 |
| FP16 | ~17 GB | RTX 3090/4090, A100 |
| INT8 (Q8_0) | ~8 GB | RTX 3060 12GB, 3080, 4070 |
| INT4 (Q4_K_M) | ~5 GB | **35 потребительских GPU**, включая RTX 3060 12GB |

**Скорость на Q4_K_M:**
- RTX 4090: ~131 tok/s
- RTX 3090: ~122 tok/s
- RTX 3060 12GB: ~47 tok/s

Для нас 47 tok/s — **достаточно** для одного NPC. Один ответ NPC — 100 токенов = ~2 секунды.

### 2.2. Реалистичные сценарии запуска

**Минимальный (пилот, один NPC):**
- RTX 3060 12GB (б/у — 25-30 тыс рублей)
- Qwen2.5-7B-Instruct в Q4_K_M
- ~5 GB VRAM на модель + 1-2 GB на контекст и активации

**Комфортный (разработка):**
- RTX 4070 / RTX 3090 12-24 GB
- FP16 или Q8_0
- ~8-17 GB VRAM
- Можно держать модель в FP16, что лучше для активационных интервенций

**Серверный (десятки NPC):**
- RTX 4090 / A100
- Batch inference (несколько NPC одновременно)
- 24+ GB VRAM

### 2.3. **Критическое ограничение**, которое надо проверить

Все рабочие реализации Persona Vectors показаны на **FP16** или **полной точности** моделях. **Не нашёл подтверждения**, что persona vectors работают на квантованных (INT4) моделях.

**Почему это может не работать:**
- Квантование меняет распределение активаций
- Активации становятся менее точными
- Тонкое управление через инжекцию маленького вектора может «теряться в шуме»

**Это надо проверять эмпирически** — это первый эксперимент, который мы должны провести.

**Если не работает на INT4 — варианты:**
- Использовать Q8_0 (8 GB VRAM, всё ещё реалистично на потребительском железе)
- Использовать FP16 на видеокартах с 16-24 GB
- Применить «усиление» — больший коэффициент α при квантовании

Это не блокер, но это **первая важная техническая неизвестная**.

## 3. Альтернативы и упрощения

### 3.1. llm_steer (`Mihaiii/llm_steer`)

Более простая библиотека, чем Persona Vectors. Использует **тексты** вместо контрастных пар:

```python
steered_model.add(layer_idx=20, coeff=0.4, text="logical")
```

То есть просто говорим «двигай в сторону концепта logical», и она извлекает вектор из этого слова. Менее точно, но быстрее в прототипировании.

**Это хороший вариант для первого спринта.** Можно начать с llm_steer для проверки концепции, потом переходить на полный Persona Vectors пайплайн для качества.

### 3.2. Уровень PyTorch hooks

Если что-то не работает в готовых либах — можно работать прямо на уровне PyTorch hooks. Это 50-100 строк кода. Полная свобода. Это уровень, на котором мы можем делать **множественные одновременные интервенции** для разных параметров характера.

## 5. Что нужно проверить эмпирически (первые эксперименты)

Прежде чем строить архитектуру, нужно ответить на эти вопросы экспериментами:

### Эксперимент 1: Persona Vectors на квантованной Qwen2.5-7B

**Цель:** проверяет ли что инжекция работает на Q8_0 / Q4_K_M.

**Метод:** взять готовый persona vector для черты «evil» из репо, применить к Qwen2.5-7B-Instruct в разных квантизациях, измерить разницу в поведении.

**Решает:** какое железо реально подходит проекту.

### Эксперимент 2: Множественные одновременные векторы

**Цель:** проверить что несколько разных векторов можно накладывать без интерференции.

**Метод:** черты «honest», «brave», «caring» — применить одновременно, измерить что NPC проявляет все три.

**Решает:** наш Принцип 2 (характер как множество параметров) — реалистичен ли он архитектурно.

### Эксперимент 3: Динамическое изменение векторов

**Цель:** проверить что коэффициенты α можно менять в реальном времени без перезапуска модели.

**Метод:** во время одного диалога постепенно увеличивать коэффициент «fear», смотреть как меняется поведение.

**Решает:** работает ли наша идея «эмоциональные параметры влияют на текущую генерацию».

### Эксперимент 4: Latency

**Цель:** измерить реальное время отклика NPC.

**Метод:** замерить от «пользователь отправил сообщение» до «получил ответ» при разных конфигурациях.

**Решает:** реалистична ли архитектура для real-time игрового цикла.

Все четыре эксперимента — **по 1-2 часа каждый**. Можно сделать в одну-две сессии.

## 7. Что мы НЕ нашли (известные неизвестные)

Эти вопросы остаются открытыми после разведки:

1. **Persona Vectors в долгосрочной симуляции.** Все эксперименты — короткие диалоги. Как векторы ведут себя через дни/недели работы — нет данных.

2. **Динамическое формирование векторов из опыта NPC.** Anthropic делает векторы из контрастных промптов (заранее). Никто не показал «формирование вектора из жизни агента». Это **наша задача**.

3. **Связь векторов с памятью.** Как именно эпизоды из памяти должны менять коэффициенты векторов — нет работающих примеров.

4. **Сон-консолидация в коде.** Принцип 3 — концептуально ясен, но **никто не реализовал**. Будем первыми.
