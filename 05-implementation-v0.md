## Часть 1. Что я узнал, прочитав код Persona Vectors

### Хорошие новости (приятные сюрпризы)

**Код проще, чем казался.** Ключевой класс `ActivationSteerer` — около 130 строк. Файл `activation_steer.py`. Основной механизм:

```python
def _hook_fn(self, module, ins, out):
    steer = self.coeff * self.vector
    # добавляем вектор к выходу слоя
    new_out = out + steer
    return new_out

# регистрируем hook на конкретный слой
self._handle = layer.register_forward_hook(self._hook_fn)
```

Это всё. Десять строк реальной логики.

**Множественные интервенции уже есть.** Класс `ActivationSteererMultiple` принимает список `instructions`, где каждая описывает один вектор + слой + коэффициент:

```python
instructions = [
    {"steering_vector": love_vec, "coeff": 1.5, "layer_idx": 20},
    {"steering_vector": fear_vec, "coeff": 0.3, "layer_idx": 18},
    {"steering_vector": brave_vec, "coeff": -0.5, "layer_idx": 22},
]
with ActivationSteererMultiple(model, instructions):
    output = model.generate(...)
```

**Это прямо то, что нам нужно** для реализации Принципа 2 (характер как множество мелких параметров). Готовая инфраструктура.

**Три варианта применения:**
- `positions="all"` — на всё подряд
- `positions="prompt"` — только при обработке промпта
- `positions="response"` — только при генерации ответа

Это даёт тонкое управление: можно одну черту применять только к восприятию (как NPC видит ситуацию), другую — только к генерации (как формулирует ответ).

**Векторы хранятся как `[layers × hidden_dim]`** — отдельный вектор для каждого слоя. Это значит **выбор слоя** — отдельный гипер-параметр. В коде по умолчанию используется `layer 20` для Qwen2.5-7B (из 28 слоёв).

**Векторы извлекаются три типа:**
- `prompt_avg` — среднее по всем токенам промпта
- `prompt_last` — только последний токен промпта  
- `response_avg` — среднее по токенам ответа

Можно использовать разные типы для разных целей.

**Мониторинг** через проекцию активаций на вектор — буквально `(activation * vector).sum(dim=-1) / vector.norm()`. Одна строчка, считается на лету.

### Ограничения реального кода

**Зависимости тяжёлые:** `unsloth==2025.5.9`, `vllm==0.8.5.post1`, `bitsandbytes==0.45.5`, `torch==2.6.0`, `transformers==4.52.3`. Это много, но всё стандартное.

**Контрастные данные генерируются через Claude API.** Скрипт `data_generation/prompts.py` показывает: для каждой черты Anthropic просили Claude-3.7-Sonnet сгенерировать 5 пар инструкций + 40 вопросов + промпт оценки. Это **внешняя зависимость** в pipeline — для генерации данных нужна большая модель. Но **сами векторы** работают локально.

**Оценка качества вектора через GPT-4.1-mini** как судью. Это снова внешняя зависимость на этапе валидации, но не на инференсе.

## Часть 2. Что я нашёл по улучшениям

После прочтения кода у меня было несколько вопросов, и поиск дал ответы. Самое важное в порядке значимости:

### Улучшение 1: Динамические steering vectors (SADI, ICLR 2025)

**Главная критика статичных векторов:** один фиксированный вектор не адаптируется к семантике входа. Если у нас «вектор любви», но контекст разговора — про работу, вектор может **ухудшать** генерацию, не улучшать.

**SADI (Semantics-Adaptive Dynamic Intervention):**
- Извлекает разности активаций из контрастных пар (как Persona Vectors)
- **Создаёт бинарную маску** — какие нейроны/головы внимания **критичны** для черты, какие нет
- Применяет интервенцию **только на критичные элементы**, остальные не трогает
- **Адаптивно масштабирует** на основе семантики текущего входа

**Что это значит для нас:** статичный «вектор любви» может ломать генерацию когда кузнец занят чисто практическим делом (ковка). Динамический вектор **усиливается** когда в контексте появляется жена, **ослабляется** когда контекст про работу. Это ровно то, как работают эмоции у живых людей.

### Улучшение 2: Семантический банк векторов (Dynamic Multimodal Activation Steering)

Подход: вместо одного вектора — **база похожих векторов**, выбор по семантической близости к текущему входу.

**Применение к нам:** черта «страх» имеет **разные подвиды** — страх за себя, страх за близких, страх неизвестного, страх перемен. Если мы храним **банк векторов страха** разных контекстов, и выбираем самый подходящий по cosine similarity к текущей ситуации — получаем гораздо более тонкое поведение.

### Улучшение 3: Контрастное декодирование (System Prompt Strength, январь 2026)

Идея: **не нужен полный доступ к активациям**. Можно достичь похожего эффекта через декодирование с двумя проходами:
- Первый проход: генерация с целевым системным промптом
- Второй проход: генерация с базовым промптом
- Финальные токены: `target_logits - α * baseline_logits`

**Что это значит:** если активационные интервенции окажутся проблемными на квантованных моделях, **есть запасной вариант** — контрастное декодирование. Стоит примерно 2× больше FLOPs, но работает с любым inference engine.

### Улучшение 4: Главное ограничение, которое надо знать

Из статьи о CAA: **«Activation Addition only manipulates internal activation patterns, but not goals, rewards and policies.»**

> «When activation addition succeeds, it is because the target behavior is already encoded in the model's residual stream in a relatively linear and disentangled way. In other words, the model already knows how to behave differently, and ActAdd merely biases the internal state toward one of those pre-existing modes.»

**Перевод на наш язык:** activation steering **не создаёт** новое поведение. Оно **усиливает** уже существующие в модели «режимы». Если в Qwen2.5 нет режима «средневековый кузнец», вектор не сделает её таковым. Но **базовые человеческие черты** (доброта, страх, любовь, жадность) у современных моделей **уже есть**, и активационные векторы успешно их усиливают.

**Это важное архитектурное ограничение.** Оно означает что **роль персонажа** (кузнец, маг, бандит) надо задавать через **промпт + окружение**, а **эмоциональное и моральное состояние** через **активационные векторы**.

### Улучшение 5: Oversteering — реальная проблема

Цитата: **«excessive emphasis on the instruction often comes at the cost of task accuracy and can also degrade the overall quality of the generated text»**

Если делать α слишком большим, NPC становится карикатурой. «Очень любящий» кузнец будет говорить только про жену, не сможет работать. Решение в литературе — **plausibility-guided dynamic control** (Directer, 2026) — система оценивает в реальном времени, не съезжает ли модель с разумного поведения.

### Улучшение 6: Layer-aware применение

Из обзоров: **разные слои отвечают за разные аспекты**. Нижние (2-8) — базовая лексика. Средние (16-22) — концепты, поведение, **это наша зона**. Верхние (24-30) — финальная формулировка. Это значит:

- Эмоции — средние слои (~18-20)
- Стиль речи — поздние слои (~24)
- Возможно, восприятие фактов — ранние слои

Это даёт нам **архитектурный принцип**: разные параметры характера применяются на разных слоях. Не «всё в layer 20».

| Стена | Что нашли в литературе | Как мы обходим |
|---|---|---|
| Статичные векторы плохо адаптируются | SADI, dynamic steering | Динамические α на основе текущего восприятия + памяти |

## Часть 4. Наша архитектура v0

Теперь, имея реальные данные, могу предложить конкретную архитектуру. Не «концептуальную картинку», а **код, который можно начать писать**.

### Структура одного NPC

```python
@dataclass
class NPC:
    # Идентичность (статичная или меняется крайне редко)
    name: str
    role: str  # "blacksmith", "guard", "merchant"
    role_prompt: str  # развёрнутое описание роли через промпт
    
    # Параметры характера (множество маленьких векторов)
    character_traits: Dict[str, TraitState]
    # ↓
    # TraitState:
    #   vector: torch.Tensor  # извлечённый вектор
    #   layer_idx: int        # на каком слое применять
    #   base_coeff: float     # базовый коэффициент (постоянный)
    #   modulation: float     # текущая модуляция (от 0 до ∞)
    #   contexts: List[str]   # семантические контексты, в которых усиливается
    
    # Эмоциональное состояние (динамическое, меняется быстро)
    emotional_state: Dict[str, float]
    # пример: {"fear": 0.2, "love": 0.8, "anger": 0.0}
    # каждое — модулирует соответствующий trait_state
    
    # Память (отдельная подсистема)
    memory: MemorySystem
    
    # Привязанности (паттерны из Принципа 2)
    attachments: Dict[str, AttachmentPattern]
    # пример: "wife_maria" → паттерн из 30 параметров
```

### Как это применяется на инференсе

```python
def npc_speak(npc: NPC, context: str, perception: Perception) -> str:
    # 1. Обновляем эмоциональное состояние от восприятия
    npc.emotional_state = update_emotions(npc, perception)
    
    # 2. Собираем активные интервенции
    instructions = []
    for trait_name, trait in npc.character_traits.items():
        # Динамический коэффициент = базовый * модуляция * семантическая близость
        dynamic_coeff = (
            trait.base_coeff 
            * npc.emotional_state.get(trait_name, 1.0)
            * semantic_match(context, trait.contexts)
        )
        # Защита от oversteering
        dynamic_coeff = clip(dynamic_coeff, -3.0, 3.0)
        
        if abs(dynamic_coeff) > 0.05:  # не применяем слабые
            instructions.append({
                "steering_vector": trait.vector,
                "coeff": dynamic_coeff,
                "layer_idx": trait.layer_idx,
                "positions": "all",
            })
    
    # 3. Готовим промпт (роль + контекст + память)
    prompt = build_prompt(npc, context, perception, npc.memory.retrieve(context))
    
    # 4. Применяем активационные интервенции
    with ActivationSteererMultiple(model, instructions):
        response = model.generate(prompt, ...)
    
    # 5. Записываем взаимодействие в память
    npc.memory.record(context, response, npc.emotional_state)
    
    # 6. Мониторим характер через проекцию (для метрик)
    measure_trait_expression(model, response, npc.character_traits)
    
    return response
```

### Что здесь принципиально нового

**1. Множественные одновременные векторы** для разных черт.

**2. Динамические коэффициенты**, зависящие от:
- Базового значения (статичный характер)
- Эмоционального состояния (меняется от событий)  
- Семантической близости (релевантность контексту)

**3. Layer-aware распределение** — разные черты на разных слоях.

**4. Защита от oversteering** через clip коэффициентов.

**5. Мониторинг проявления** через проекцию — для метрик и диагностики.

**6. Связка с памятью** — память влияет на промпт + эмоциональное состояние.

## Часть 6. Что мы пока не знаем (честно)

После всей работы остаются открытые вопросы. Не блокеры, но риски:

**Quантование** — главный риск. Если на Q4 не работает, требования к железу растут.

**Долгосрочная работа** — все эксперименты в литературе короткие. Что будет с NPC через 100 часов жизни — никто не знает.

**Русский язык** — Qwen2.5 умеет русский, но persona vectors извлекались на английских данных. Возможно, для русских NPC нужны векторы извлечённые на русских контрастных данных.

**Сон-консолидация** — ключевой Принцип 3 — как именно реализовать модификацию векторов после «сна»? Это **никто не делал**. Есть несколько идей:
- Считать новые векторы из накопленного опыта дня
- Линейно комбинировать старые и новые с весом
- Корректировать через RL подобный механизм

Это будет **наш реальный исследовательский вклад**.

**Поток мыслей** (Принцип 4) — отдельная архитектурная задача, не покрывается активационным управлением. Inner Thoughts framework даёт скелет, но интеграция с нашей системой — отдельная работа.
